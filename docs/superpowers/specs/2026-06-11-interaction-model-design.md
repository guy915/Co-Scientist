# Interaction Model — Design Spec

**Date:** 2026-06-11
**Status:** Approved

## Problem

The Co-Scientist app is currently fire-and-forget: once a run starts, the user can only watch or cancel. There is no way to steer the research direction mid-run, ask the system questions about what it's doing, or receive proactive updates in a conversational form.

## Goal

Add a bidirectional interaction model that lets the user:
1. **Steer** the running system — queue guidance that shapes the next evolution cycle
2. **Ask questions** — get answers about the live run state from a fast LLM
3. **Receive milestone updates** — auto-generated messages at key agent events

The running LangGraph graph is never interrupted. The original research goal is never modified.

---

## Approach: App-layer overlay (no engine changes)

All new logic lives in the FastAPI app layer (`ai-coscientist-app/`). The `ai-coscientist-engine/` package is untouched.

**Steering** messages queue in SQLite and are injected as preferences text by `engine_adapter.py` before each supervisor call. The original `research_goal` field is preserved; steering — including hard redirects — is appended alongside it so the supervisor sees both.

**Q&A** is a one-shot LLM call (configurable fast/cheap model) with full run context. No interruption to the workflow.

**System messages** are auto-generated by `engine_adapter.py` at key node completions and stored alongside user messages.

---

## Data Layer

### New table: `messages`

Added to `store.py` alongside the existing schema.

```sql
CREATE TABLE messages (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id     TEXT NOT NULL,
    sender     TEXT NOT NULL,   -- 'user' | 'system'
    content    TEXT NOT NULL,
    kind       TEXT NOT NULL,   -- 'steering' | 'qa' | 'milestone'
    created_at TEXT NOT NULL,
    applied    INTEGER DEFAULT 0  -- 1 once steering message injected into supervisor
);
```

`id` is the ordering key. `GET /api/runs/{id}/messages` returns rows ordered by `id ASC`.

Messages persist indefinitely. The existing `events` table is unchanged — chat messages have a separate lifecycle (readable after run ends, not tied to SSE replay).

### Store functions (add to `store.py`)

- `append_message(run_id, sender, content, kind) -> Message`
- `list_messages(run_id) -> List[Message]`
- `get_pending_steering(run_id) -> List[Message]` — `kind='steering' AND applied=0`
- `mark_steering_applied(ids: List[int])`

## Backend API — 3 new endpoints (add to `runs.py`)

```
POST /api/runs/{id}/messages
GET  /api/runs/{id}/messages
POST /api/runs/{id}/messages/ask
```

### POST /api/runs/{id}/messages

Accepts `{ content: str }`. Stores a **steering** message only. Returns immediately with `{ id, kind: "steering", status: "queued" }`.

Auto-classifies as steering by default. If the client detects a question (question words or trailing `?`), it should call `POST /messages/ask` instead. The mode indicator in the UI handles this routing.

### GET /api/runs/{id}/messages

Returns all messages for a run in sequence order. Used by the frontend to hydrate the Chat tab on mount and on poll.

### POST /api/runs/{id}/messages/ask

Accepts `{ question: str }`. Stores the question as `kind='qa'`, then makes a single LLM call using `CHAT_MODEL_NAME` (new env var, defaults to `MODEL_NAME` if unset).

**Context passed to the LLM:**
- Research goal (original, unmodified)
- Top 5 hypotheses with current Elo scores and one-line summaries
- Last iteration's review summaries (truncated to fit context)
- Last 3 match rationales from the tournament
- Last 10 messages in conversation history (for continuity)

The answer streams back as SSE (`text/event-stream`). On completion the answer is stored as `sender='system', kind='qa'`.

## Steering Mechanism

In `engine_adapter.py`, before each new iteration call to `generate_hypotheses()`:

1. Call `store.get_pending_steering(run_id)` to retrieve unread steering messages.
2. If any exist, format as:
   ```
   User steering guidance (apply to this iteration):
   - <message 1>
   - <message 2>
   ```
3. Append this block to the existing `preferences` string in the generation config.
4. After the iteration starts successfully, call `store.mark_steering_applied(ids)`.

The original `research_goal` in `WorkflowState` is never modified. Hard redirects ("pivot goal to X") work identically — they appear as strong preference guidance alongside the original goal. The supervisor naturally treats forceful language as high-priority direction while maintaining research continuity.

## System Milestone Messages

`engine_adapter.py` auto-generates `sender='system', kind='milestone'` messages at these node completions:

| Node | Message content |
|------|----------------|
| `supervisor` | `"Research plan ready — {first sentence of supervisor_guidance summary}"` |
| `generate` | `"Iteration {N}: {count} hypotheses generated"` |
| `ranking` | `"Tournament complete — {top hypothesis title} leads (Elo {score})"` |
| `meta_review` | `"Meta-review: {one-sentence insight from meta_review output}"` |
| `evolve` | `"{count} hypotheses evolved from top performers"` |

These give the user a live sense of progress in the Chat tab without switching to Overview.

## Frontend

### Chat Tab

New sixth tab labelled **"Chat"** added to `RunDetail.tsx`. Tab component: `src/workbench/components/tabs/ChatTab.tsx`.

#### Feed row types

| Type | Visual |
|------|--------|
| Milestone (system) | Teal dot · agent label · summary text · timestamp. Compact, no bubble. |
| User steering | Right-aligned bubble. Small badge: "Steering · pending" or "Steering · applied". |
| User Q&A | Right-aligned bubble, no badge. |
| Answer | Left-aligned, indented, slightly dimmer. Rendered inline below its question. |

#### Input area

- Single textarea at the bottom of the tab.
- `Ctrl+Enter` or send button to submit.
- Small mode indicator (auto-detected): shows "Steer" or "Ask" label, tappable to override.
- Disabled while the previous Q&A answer is still streaming.

#### Data fetching

New hook `src/hooks/useMessages.ts` — `useMessages(runId)`:
- Fetches `GET /api/runs/{id}/messages` on mount.
- Polls every 3 seconds while `run.status === 'running'`.
- Exposes `{ messages, sendMessage, askQuestion, isAnswering }`.

No new WebSocket or persistent SSE stream — polling is sufficient since messages arrive at iteration granularity (minutes).

## Testing

**Backend (pytest):**
- `test_messages.py`: message storage, steering injection text formatting, milestone auto-generation, Q&A endpoint (mock LLM call), `get_pending_steering` / `mark_applied` lifecycle

**Manual:**
1. Start a run, switch to Chat tab — verify milestone messages appear as supervisor/generate/ranking nodes complete
2. Send a steering message — verify it shows as "pending", then "applied" after next iteration starts, and that iteration's hypotheses reflect the guidance
3. Send a Q&A question ("why did hypothesis X rank highest?") — verify answer streams back and appears inline below the question
4. Send a hard redirect ("pivot to preventative interventions") — verify original goal still visible in Overview, but next iteration's hypotheses shift direction
5. Refresh the page — verify Chat tab rehydrates from DB with full history
